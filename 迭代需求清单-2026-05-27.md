### 需求 1：下载按钮内容有误

**所属模块**：审核报告页 —— 右下角操作区

**问题描述**：
审核结果出来后，页面底部有三个下载按钮：「保存为 PDF」「保存为图片」「保存 JSON」。用户点击后下载到本地文件，但文件内容是错误的——可能是空白、内容截断、或者根本不是审核报告的内容。

**当前实现方式**：
- 「保存为 PDF」→ 调用浏览器 `window.print()`，依赖打印样式
- 「保存为图片」→ 使用 `html-to-image` 库的 `toPng()` 截取报告 DOM
- 「保存 JSON」→ 将 `ComplianceReport` 对象序列化为 JSON 文件下载

**可能原因**：报告组件 `ReceiptReport` 使用了分页机制（`ITEMS_PER_PAGE = 4`），问题卡片超过 4 条会分页。下载时可能只截取了当前页，导致内容不完整。另外导出图片时 `pixelRatio: 2` 可能导致截取范围溢出或空白。

**怎么修**：
- PDF 下载：在打印前切换到"全部显示"模式，确保所有问题在一页内
- 图片下载：同样先展平分页，再截图；或改为全量渲染后截取整个报告区域
- JSON 下载：检查 `sessionStorage` 取出的 `complianceReport` 是否完整

### 需求 2：包装物理尺寸填写后未参与审核

**所属模块**：分析页 —— 左侧面板

**问题描述**：
用户在分析页左侧可以填写包装的物理宽度和高度（mm 单位），这个参数的本意是让 AI 检查包装上的文字高度是否 ≥ 1.8mm（GB 7718 强制要求）。但目前填写后，审核结果中完全没有体现这个检查项。

**当前实现方式**：
```ts
// 前端：analyze/page.tsx
body.physicalWidth = physicalWidth || null
body.physicalHeight = physicalHeight || null
```
前端确实把尺寸传给了 API，但 API 路由中的提示词没有要求 AI 用这个数据做文字高度校验，AI 自然不输出。

**怎么修**：
在审核提示词中增加一段：如果提供了物理尺寸，则根据图片分辨率换算，检查包装上所有强制标注内容的文字高度是否 ≥ 1.8mm，将不符合的标注为错误。

## 二、UI 布局优化（1 条）

---

### 需求 3：审核流程步骤条文字被裁切

**所属模块**：分析页 —— 左侧步骤条

**问题描述**：
步骤条从「视觉识别提取」到「生成审核报告」共四步，每步都有标题和一行描述文字（如第 1 步的 "Doubao-Vision-Pro-32K 提取文字"）。因为这条描述文字较长，而每个步骤分配的横向空间有限，导致文字在边缘被截断。

**当前样式**：
```css
.stepWrapper { flex: 1; min-width: 0; }
.stepLabel { font-size: 14px; white-space: nowrap; }
.stepDesc { font-size: 12px; white-space: nowrap; }
```
`flex: 1` 让四步均分左面板宽度，`white-space: nowrap` 导致长文字不换行直接溢出。

**怎么修**：
- 方案 A：允许描述文字换行（去掉 `white-space: nowrap`），增加步骤条纵向间距
- 方案 B：把左面板再略微加宽，让 nowrap 文字能完整放下
- 推荐 A+B 组合：允许换行 + 面板宽度适当放大

## 三、审核提示词优化（4 条）

---

### 需求 4：审核结果按类别分块输出（格式塔亲密性）

**所属模块**：审核 API 提示词（analyze + analyze-kimi）

**问题描述**：
目前 AI 返回的审核结果中，产品名称、配料表、供应商名称、委托商地址、执行标准号等信息是混合排列的。例如第 1 条是产品名，第 2 条跳到供应商电话，第 3 条又回到净含量——信息没有按"同类相聚"的原则组织，阅读体验差。

**期望输出分组**：

审核结果按以下 6 大类分块输出，同类信息紧密排列，异类之间留明显分隔：

| 分组 | 包含内容 |
|------|----------|
| 品牌名称 | 品牌、商标 |
| 基本信息 | 产品名称、产品类型、规格/净含量、配料表 |
| 生产方信息 | 生产商/供应商名称、地址、电话/邮编 |
| 委托方信息 | 委托商名称、地址、电话/邮编（如有） |
| 执行标准 | 产品标准号、SC 许可证号、食品生产许可证编号 |
| 营养成分 | 营养成分表、能量值及营养素数值逻辑 |
| 其他信息 | 贮存条件、保质期、生产日期、致敏原、食用方法 |

提示词中强制要求 AI 按此分组结构输出，并规定同组条目之间不插入其他组内容。

**怎么修**：
修改 `app/api/analyze/route.ts` 和 `app/api/analyze-kimi/route.ts` 中的 `system prompt`，在"输出 JSON 格式"部分加入上述分组定义。

### 需求 5：法规条款引用 —— 悬停弹窗 / 点击弹窗查看原文

**所属模块**：审核报告展示（ReceiptReport）

**问题描述**：
目前审核结果中有一条 `regulation` 字段，写明了违法/违规依据，例如 `"GB 7718 第 4.1.2 条"`。但这是纯文字，用户无法直接查看条款原文。跳转到新页面打开 PDF 路径太长、体验不好。

**期望效果（按优先级）**：

**方案 A（首选）：鼠标悬停 → 小弹窗显示条款原文**
- 法规条款文字显示为蓝色可交互样式
- 鼠标悬停时，旁边浮出一个小弹窗（tooltip/popover）
- 弹窗内显示该条款的原文内容，不需要打开任何新页面
- 弹窗尺寸适中（~400px 宽），内容可滚动

**方案 B（备选）：点击 → 模态弹窗展示文档并自动定位**
- 点击法规条款文字后，弹出模态窗口（modal）
- 窗口内嵌入对应 PDF 文档的指定页码
- 自动滚动/跳转到该条款所在的段落

**实现路径**：

由于直接从 PDF 实时提取指定条款难度较大，推荐方式为：**预处理国标为结构化数据**。

```
1. 将 GB 7718 / GB 28050 / GB 2760 三部国标的关键条款
   提取为结构化 JSON：
   {
     "GB 7718": {
       "4.1.2": "预包装食品的标签应当...（条款原文）",
       "4.1.3": "..."
     },
     "GB 28050": { ... },
     "GB 2760": { ... }
   }

2. 存放在 public/knowledge_base/clauses.json

3. AI 返回 regulation 时附带 regulationDoc 和 regulationClause
   例如：{ "regulationDoc": "GB 7718", "regulationClause": "4.1.2" }

4. 前端 ReceiptReport 根据这两个字段查询 clauses.json 获取原文
   - 方案 A：hover 时用 Tooltip 组件展示
   - 方案 B：click 时用 Modal 组件展示
```

**技术可行性**：
- 三部国标中每条规范约 100-500 字，JSON 文件整体 < 200KB
- 方案 A（hover tooltip）不需要额外依赖，纯 CSS + React state 即可
- 方案 B（click modal）可复用已有的 DocumentModal 组件
- 人工整理 + AI 辅助提取条款，1-2 小时可完成结构化

**怎么修**：
1. 新建 `public/knowledge_base/clauses.json`：三部国标的条款结构化数据
2. 提示词中增加字段：`regulationDoc`（文档名）、`regulationClause`（条款号）
3. 前端 `ReceiptReport` 新增 `RegulationPopover` 组件：
   - 优先实现方案 A（hover 弹窗）
   - 若方案 A 实现困难，降级为方案 B（click modal）

### 需求 6：计量称重产品的净含量豁免

**所属模块**：审核 API 提示词

**问题描述**：
有些食品是散装称重/计量称重销售的（比如散装糖果、称重坚果），包装上会标注"计量称重"而非具体的净含量数值。但 OCR 识别后，AI 发现没有净含量数字就直接报错"净含量缺失"——这属于误报，因为计量称重产品确实不需要标注净含量。

**期望行为**：
- AI 识别到"计量称重""散装称重""称重销售"等关键词时
- 不将"缺少净含量数值"列为错误
- 改为在结果中增加一条备注："该产品为计量称重销售，依据 GB 7718 无需标注净含量"

**怎么修**：
在审核提示词中增加一段专项规则：先检查是否标注了计量称重，若是，则跳过净含量校验并添加豁免备注。

### 需求 7：检测英文翻译准确性

**所属模块**：审核 API 提示词

**问题描述**：
很多食品包装上带有英文标识（产品名、配料翻译、营养信息等），但翻译可能存在错误、语法问题或误导性表述。目前审核流程完全不检查英文内容。

**期望行为**：
- AI 识别到包装上的英文内容后
- 检查：拼写是否正确、中文与英文是否对应（翻译准确性）、是否有语法错误
- 如果英文声称某种功效（如 "low fat"、"organic" 等），检查是否符合国内法规对应定义
- 将发现的问题列入审核报告

**怎么修**：
在审核提示词中增加英文检测规则段落，作为营养成分和产品名称审核的补充检查项。

### 需求 8：用户反馈入口

**所属模块**：审核报告页 / 新增 API

**需求背景**：
用户收到 AI 审核结果后，可能发现 AI 遗漏了某个问题、误判了某项内容、或给出了错误的修改建议。目前没有任何途径让用户把这些纠正信息反馈回来。没有反馈，AI 就无法知道自己错了，也无法改进。

**功能描述**：
在审核报告页右下角（下载按钮旁边）增加一个「反馈」按钮。点击后弹出简单表单：

```
┌─────────────────────────────┐
│  问题反馈                    │
│                             │
│  问题类型：                  │
│  ○ 审核遗漏  ○ 审核错误      │
│  ○ 建议补充  ○ 其他          │
│                             │
│  问题描述：                  │
│  ┌─────────────────────┐    │
│  │                     │    │
│  └─────────────────────┘    │
│                             │
│  [取消]  [提交反馈]          │
└─────────────────────────────┘
```

**怎么修**：
- 前端：`ReceiptReport` 组件新增反馈按钮 + 弹窗（React state 控制）
- 后端：新增 `/api/feedback` 路由，接收反馈数据
- 存储：暂时写入 Upstash Redis（项目已有），或先写入 Vercel 的简易 KV

**依赖**：需要需求 9（数据沉淀）的基础设施。

### 需求 9：后端数据沉淀与 AI 自我进化

**所属模块**：新增后端数据层

**需求背景**：
用户每次审核都会产生一批数据：原始图片（base64）、OCR 提取文字、AI 分类结果、AI 审核报告、用户反馈。目前这些数据没有任何留存——审核完毕就没了，无法回溯、无法分析、无法用来优化 AI。

**功能描述（分两阶段）**：

**阶段一：数据记录**
- 每次审核完成后，将以下数据打包存入后端：
  - 审核 ID + 时间戳
  - 原始 OCR 文字
  - AI 分类结果
  - AI 审核报告（完整 JSON）
  - 用户反馈（如有）
- 提供管理后台页面，可查看历史审核记录

**阶段二：AI 进化**
- 定期汇总用户反馈，提取高频遗漏项和误判项
- 将这些模式写回审核提示词，让 AI 在后续审核中自动规避已知错误
- 例如：过去 10 次中有 4 次反馈"遗漏了 SC 证号格式校验"，则提示词中增加"请重点检查 SC 证号是否为 SC+14 位数字"

**待讨论**：
- 存储方案：Upstash Redis（已有）→ 免费额度够用，但需定期归档；或接入 Supabase / PlanetScale 做关系存储
- 数据分类：按哪些维度归档？（按时间/按产品类型/按问题类型）
- 是否需要新增一个「数据看板」页面？

**怎么修**：
- 新建 `/api/history` 路由：POST 写入审核记录，GET 查询历史
- 新建 `/app/history/page.tsx`：历史记录页
- 本次迭代先做阶段一（记录），进化逻辑后续迭代

### 需求 10：首页新增「经验库」模块

**所属模块**：首页知识库区域

**需求背景**：
首页左侧知识库目前有：
- 3 张国标卡片（GB 28050 / GB 7718 / GB 2760）
- 行业标准占位卡片
- 广告法禁用词卡片
- 品牌规范上传卡片

这些都是"外部知识"——国家标准、行业规定、品牌给的文档。但用户自己在审核过程中也会积累经验（比如"某类产品常见的配料标注错误""某厂商经常漏标致敏原"等）。这些经验目前无处存放。

**功能描述**：
在首页左侧知识库区域新增一张卡片「我的经验」，与现有的国标卡片并列展示。点击后进入经验管理页，支持：

- 新增经验条目（标题 + 分类标签 + Markdown 正文）
- 编辑已有条目
- 列表浏览所有经验
- 在审核时，将经验库内容作为附加上下文传入 AI（类似目前的品牌文件上传逻辑）

```
首页卡片布局：
┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐
│ GB28050 │ │ GB7718  │ │ GB2760  │ │ 行业标准│ │ 广告法  │ │ 品牌规范│
└─────────┘ └─────────┘ └─────────┘ └─────────┘ └─────────┘ └─────────┘
                                                          ┌─────────┐
                                                          │ 我的经验 │  ← 新增
                                                          └─────────┘
```

**怎么修**：
- 新建 `/app/experience/page.tsx`：经验管理页（新增 / 编辑 / 列表）
- 新建 `/api/experience/route.ts`：CRUD 接口
- 首页 `home/page.tsx` 新增一张 TiltedCard 卡片
- 审核页 `analyze/page.tsx` 分析和分类时将经验上下文传入

**依赖**：需要需求 9 的数据存储方案。
