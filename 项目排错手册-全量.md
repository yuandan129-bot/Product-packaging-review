## 一、Vercel 部署与构建

### 1.1 Root Directory 错误

**现象**：Vercel 构建失败，提示 `No Next.js version detected`。

**根因**：项目曾在子目录 `packaging-review-app/` 中，Vercel 默认读根目录的 `package.json`，找不到 `next` 依赖。

**解决**：
- 方案 A（已采用）：将项目从子目录提至根目录，`package.json` 直接位于仓库根目录
- 方案 B：Vercel Settings → General → Root Directory 设为 `packaging-review-app`

**已修复**：项目已扁平化，无需额外设置。

---

### 1.2 环境变量保存后不生效

**现象**：在 Vercel 添加环境变量后部署，运行时仍读取不到。

**根因**：Vercel 环境变量保存 ≠ 注入，**必须触发一次新的部署**才能生效。

**解决**：Settings → Environment Variables → Save → 回到 Deployments → 点击最新部署的 `⋯` → **Redeploy**。

**记忆口诀**：保存变量 → 触发部署 → 等待完成 → 再测试。

---

### 1.3 环境变量 Scope 未勾选 Production

**现象**：本地能读到变量，Vercel 上始终读不到。

**根因**：Vercel 环境变量有三个 scope：Production / Preview / Development。如果只勾了 Development，生产环境读不到。

**解决**：添加变量时确保 **Production** 勾选。如果已有变量未生效，点击编辑检查 scope。

---

### 1.4 旧部署缓存

**现象**：push 了新代码但 Vercel 显示的还是旧版本。

**根因**：Vercel Git 集成失效（重新初始化 git 后可能断开）。

**解决**：Vercel 项目 → Settings → Git → 确认 GitHub Repository 仍绑定正确 → 如果断开，重新连接。也可手动 `git push` 触发新的 Production Deployment。

## 二、AI 模型接入

### 2.1 模型 ID 与控制台展示名不一致（豆包特有问题）

**现象**：API Key 已验证通过，但调用报 404 `model X does not exist`。

**根因**：火山引擎控制台展示的模型名（如 `Doubao-1.5-vision-pro-32k`）与 API 实际 model ID（`doubao-1-5-vision-pro-32k-250115`）**格式不同**——控制台用点号，API 用连字符 + 日期后缀。

**解决**：**必须以官方 API 文档或 curl 示例中的 model 字段为准**，不看控制台展示名。

| 来源 | 示例 | 是否可用 |
|------|------|----------|
| 控制台展示名 | `Doubao-1.5-vision-pro-32k` | ❌ |
| API 实际 ID | `doubao-1-5-vision-pro-32k-250115` | ✅ |

**通用原则**：接入任何新模型时，第一步是跑通官方 curl 示例，确认 model ID 后再写入代码。

---

### 2.2 豆包 vs DeepSeek/Kimi 接入差异

**现象**：以为所有 AI API 接入方式一样，直接复用 DeepSeek 的配置模式。

**根因**：不同平台 API 规范不同。

| | DeepSeek / Kimi | 豆包（火山引擎 Ark） |
|---|---|---|
| Key 前缀 | `sk-` | `ark-` |
| 端点 | `api.deepseek.com` | `ark.cn-beijing.volces.com` |
| 模型 ID | 文档直接给，简短 | **必须复制 curl 示例中的完整 ID** |
| 接入复杂度 | 拿到 Key 即用 | 需要确认模型 ID + 端点 + 区域 |

---

### 2.3 OCR 引擎选型 — Tesseract 精度不足

**现象**：Tesseract.js 浏览器端 OCR 识别中文包装标签准确度低，漏字错字严重。

**根因**：Tesseract 是通用 OCR 引擎，对食品包装这种密集排版、多字体混排、小字标注的场景精度不够。

**解决**：将 OCR 识别步骤从本地 Tesseract.js 替换为云端视觉 LLM（豆包 Vision Pro 32K）。流程变为：图片上传 → 豆包 API 识别文字 → DeepSeek 分类 → DeepSeek/Kimi 审核。

---

### 2.4 API 路由 Runtime 不一致

**现象**：新加的 API 路由用了 `edge` runtime，但项目其他路由用默认 Node.js runtime，导致 `process.env` 取值行为不一致。

**根因**：Edge Runtime 对 `process.env` 的处理方式与 Node.js Runtime 不同，可能导致环境变量读取异常。

**解决**：统一使用默认 Node.js Runtime（不声明 `export const runtime = 'edge'`），与项目其他 API 路由保持一致。

## 三、Git 与代码管理

### 3.1 嵌套 Git 仓库冲突

**现象**：执行 `git add` 时提示 `adding embedded git repository`，子目录被当做一个整体 blob 而非展开文件。

**根因**：项目根目录和 `packaging-review-app/` 目录各有一个 `.git`，形成嵌套仓库。

**解决**：删除子目录的 `.git`（`rm -rf packaging-review-app/.git`），从根目录统一管理。

**预防**：克隆项目后检查是否只有根目录一个 `.git`。

---

### 3.2 GitHub Push 失败 — HTTP/2 协议兼容

**现象**：`git push` 报 `Error in the HTTP2 framing layer` 或 `Recv failure: Operation timed out`。

**根因**：国内网络环境对 HTTP/2 协议支持不稳定，连接 GitHub 时 HTTP/2 帧解析失败。

**解决**：强制 Git 使用 HTTP/1.1：
```bash
git config http.version HTTP/1.1
```

**预防**：此配置已全局生效，后续无需重复操作。

---

### 3.3 GitHub Push 失败 — 网络超时

**现象**：`git push` 报 `Failed to connect to github.com port 443`。

**根因**：国内网络对 GitHub 直连不稳定，常被间歇性阻断。

**解决**：
- 开启 VPN/代理，然后推送
- 若使用代理，可配置 `git config http.proxy http://127.0.0.1:代理端口`
- 换网络环境重试

---

### 3.4 Rebase 冲突 — Push 被拒

**现象**：`git push` 被拒，提示 `fetch first`，pull 后出现 rebase 冲突。

**根因**：远程仓库有本地不存在的提交（比如之前从另一台机器 push 过）。

**解决**：
```bash
git fetch origin main
git rebase origin/main
# 解决冲突文件（打开文件删掉 <<<  >>> === 标记，保留需要的内容）
git add <冲突文件>
git rebase --continue
git push
```

**预防**：Push 前先 `git fetch` 确认远程是否有新提交。

---

### 3.5 Git Remote 丢失

**现象**：新的 `.git` init 后 `git remote -v` 为空，找不到远程仓库。

**根因**：重新 `git init` 或删除旧 `.git` 后，remote 配置丢失。

**解决**：
```bash
git remote add origin https://github.com/yuandan129-bot/Product-packaging-review.git
git push -u origin main
```

## 四、项目结构与文件管理

### 4.1 根目录陈旧副本污染

**现象**：项目同时存在 `app/`、`components/` 两套代码（根目录一套旧版，`packaging-review-app/` 一套新版），新旧不一致导致 Vercel 读到旧版。

**根因**：旧仓库结构残留——根目录的旧 `app/components/public` 等是迁移前的副本，缺少后续新增的 `ocr-doubao` 等路由。

**解决**：删除所有根目录陈旧副本，只保留 `packaging-review-app/` 内的真源。后进一步将项目扁平化。

---

### 4.2 缓存文件膨胀

**现象**：项目体积近 1GB，大部分是缓存。

**根因**：`node_modules/`（517MB）、`.next/`（377MB）、`*.traineddata`（7.4MB）、重复的 PDF 副本（3 份）占据大量空间。

**解决**：
- 删除 `node_modules/` 和 `.next/`（可随时重建）
- 删除 Tesseract 语言包（已被豆包替代）
- 删除重复的 knowledge_base PDF 副本
- 删除所有 `.DS_Store`

**已修复**：清理后释放约 915MB。

---

### 4.3 废弃依赖未清理

**现象**：`package.json` 中保留了不再使用的 `tesseract.js` 依赖。

**根因**：替换技术方案后只更新了代码，忘了清理依赖。

**解决**：OCR 从 Tesseract 改为豆包后，从 `package.json` 移除 `tesseract.js`。

**预防**：替换任何技术方案时，同步检查依赖是否需要移除。

## 五、UI 与交互 Bug

### 5.1 图片上传需要点击两次

**现象**：首页点击上传圆圈，需要点两次才能成功上传，或点击没反应。

**根因**：上传圆的 `<div onClick>` 和 `<label htmlFor>` 同时触发文件选择器，导致双弹窗——第一次选中文件被第二次弹窗覆盖。

**解决**：移除 `div` 的 `onClick`，只保留 `<label htmlFor="imageUpload">` 的原生行为。

---

### 5.2 首页刷新闪到其他页面

**现象**：刷新 `/home` 页面时，先短暂显示导航栏和标题，再跳回 splash 页。

**根因**：`useEffect` 中的 `splashSeen` 检查是异步的，在客户端渲染后才执行。检查期间页面内容已经渲染完毕，造成闪屏。

**解决**：添加 `isChecking` 加载守卫——检查完成前 `return null`，不渲染任何内容。

---

### 5.3 根路由重定向闪白

**现象**：访问根域名时短暂闪烁空白页再跳 splash。

**根因**：根 `page.tsx` 用客户端 `useEffect` + `router.push` 做重定向，React 渲染逻辑先执行了 `return null`。

**解决**：改为服务端 `redirect('/splash')`，不经过客户端渲染。

---

### 5.4 左侧面板内容被裁切

**现象**：图片上传后的分析页，左侧面板内容太长，底部按钮被裁切，无法滑动到达。

**根因**：`.splitLayout` 设置了 `height: 100vh; overflow: hidden`，而 `.splitLeft` 没有独立的高度约束和滚动。

**解决**：给 `.splitLeft` 添加 `overflow-y: auto; height: calc(100vh - 2 * var(--space-xl))`，独立滚动。

---

### 5.5 审核报告间距过大

**现象**：合规检测报告中各区块之间空白太多，信息密度低。

**根因**：报告 CSS 默认 padding / margin / divider 间距偏大。

**解决**：全局收紧 30% 的 padding、margin、line-height 和 divider 间距。

---

### 5.6 错误信息无反馈

**现象**：OCR/视觉识别失败时，错误只在右侧控制台一闪而过，用户看不到。

**根因**：控制台面板在 `isOcrRunning=false` 时自动隐藏，错误日志随之消失。

**解决**：错误处理中加入 `alert()` 弹窗，和审核失败的处理方式保持一致。同时 API 错误响应增加详细诊断信息。

---

### 5.7 OCR 品牌文案不一致

**现象**：部分 UI 文案写"OCR + DeepSeek"，部分写"豆包 + DeepSeek"，新旧混乱。

**根因**：替换 OCR 引擎后逐一修改了 UI 文案，但存在遗漏。

**解决**：全局 grep 搜索 `OCR` 和 `豆包`，统一替换为具体模型名。已统一为 `Doubao-Vision-Pro-32K`。

## 六、环境变量与安全

### 6.1 API Key 泄露风险

**出现场景**：曾将 API Key 写在文档、代码注释、commit 信息中。

**风险**：项目托管在 GitHub 公开仓库，泄露 API Key 会被滥用、产生费用。

**规则**：
- API Key **只存在于** `.env.local`
- `.env.local` 必须在 `.gitignore` 中
- `.env.example` 作为模板可提交（只含变量名不含值）
- 每次新增 Key 时，先确认 `.gitignore` 覆盖到位

---

### 6.2 本地 .env.local 修改后不生效

**现象**：修改 `.env.local` 后重启服务器，仍读到旧值或空值。

**根因**：Next.js 开发服务器在启动时一次性加载环境变量，运行期间修改 `.env.local` 不会热更新。

**解决**：每次修改 `.env.local` 后 **Ctrl+C 停掉 `npm run dev`，重新启动**。

## 附录 B：接入新 AI 模型的标准流程

1. 在平台上获取 API Key 和官方 curl 示例
2. **先跑通 curl**，确认 Key、model ID、端点都正确
3. 对比平台 API 文档，确认请求体结构（`messages` 格式、`image_url` 位置等）
4. 在项目新建 `app/api/<name>/route.ts`，复用已验证的参数
5. `.env.local` 添加 Key，`.env.example` 同步更新模板
6. 前端页面添加调用逻辑 + Loading 状态 + 错误弹窗
7. Vercel 部署后在环境变量中配置同名字段 → Redeploy → 验证

## 附录 C：项目文件清理提示词

> 每次 push 前，可将此提示词粘贴给 AI，自动审计项目中的冗余文件。

```text
在我 push 代码到 GitHub 之前，请帮我做一次项目文件清理审计。
按以下流程执行：

## 第一步：全量扫描
- 统计项目根目录及所有子目录的文件夹大小（du -sh）
- 列出所有文件（排除 .git、node_modules、.next），按目录分组
- 标注每个文件的大致用途

## 第二步：识别冗余
用以下 6 条规则逐一比对，标记可疑文件：

1. **构建缓存**：.next/、node_modules/、*.tsbuildinfo
2. **系统垃圾**：.DS_Store、Thumbs.db
3. **废弃依赖产物**：因为技术栈变更而不再使用的文件
   （如 Tesseract 语言包 .traineddata、已删除的库的残留配置）
4. **陈旧副本**：同一文件在多个目录下有副本，且内容不一致、
   缺少最新改动（用 diff 对比确认）
5. **一次性产物**：预览页、临时脚本、一次性审查报告、
   未被任何路由引用的独立 HTML
6. **冲突残留**：git rebase/merge 后残留的 .orig 文件

## 第三步：输出删除建议表
按以下格式列出所有可疑文件，标注风险等级：

| 文件 | 大小 | 分类 | 风险 | 说明 |
|------|------|------|------|------|
| xxx | xxMB | 构建缓存 | 无 | 可 npm install 重建 |
| yyy | xxMB | 陈旧副本 | 中 | 与真源不一致，缺少新路由 |

风险等级说明：
- **无**：随时可重建，删除不影响任何功能
- **低**：已确认不再使用，删除安全
- **中**：疑似无用，建议手动确认后再删
- **高**：可能有用，仅建议保留或移动

## 第四步：等待我的确认
输出表格后停止，由我来勾选哪些要删、哪些保留。
不要自行执行删除。
```
